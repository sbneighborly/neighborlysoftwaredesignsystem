Getting Started

Everything a new developer needs to install, configure, and start building with the Neighborly design system. This section covers installation, theme setup, token import, folder structure, and where to get help.

Installation

# 1. Install MUI and dependencies npm install @mui/material @mui/icons-material @emotion/react @emotion/styled # 2. Install extended component packages npm install @mui/x-data-grid @mui/x-date-pickers # 3. Install dayjs (required by DatePicker) npm install dayjs # 4. Install Nunito Sans (or load via Google Fonts — see Typography section) npm install @fontsource/nunito-sans

Theme Setup

// src/theme/index.ts import { createTheme } from '@mui/material/styles'; const theme = createTheme({ palette: { primary: { main: '#188af4', dark: '#1278d8', contrastText: '#fff' }, secondary: { main: '#00419D' }, error: { main: '#D32F2F' }, warning: { main: '#ED6C02' }, success: { main: '#3aab5c' }, }, typography: { fontFamily: "'Nunito Sans', 'Avenir Next LT Pro', -apple-system, sans-serif", button: { textTransform: 'none', fontWeight: 600 }, }, shape: { borderRadius: 6 }, components: { MuiButton: { styleOverrides: { root: { textTransform: 'none', fontWeight: 600 } } }, MuiChip: { styleOverrides: { root: { fontWeight: 600 } } }, }, }); export default theme;

Wrap Your App

// src/main.tsx (or index.tsx) import { ThemeProvider, CssBaseline } from '@mui/material'; import theme from './theme'; <ThemeProvider theme={theme}> <CssBaseline /> <App /> </ThemeProvider>

Import Design Tokens

/* In your global CSS or index.css */ @import url('https://sbneighborly.github.io/neighborlysoftwarebrandguidelines/tokens.css'); /* Then use tokens in your CSS */ .my-component { color: var(--nbl-primary); background: var(--nbl-surface); border-radius: var(--nbl-radius-md); }

Line Length

Body text should be constrained to a maximum of 65 characters per line (max-width: 65ch). This prevents eye fatigue on wide screens and keeps reading comfortable for caseworkers who spend long sessions in the application.

Easing Rationale

EasingMental modelWhen to use in Neighborly
easeOut Elements arrive — they decelerate as they settle into place Dialogs opening, drawers sliding in, dropdowns appearing, toasts entering. Anything the user is waiting to interact with should feel like it lands, not crashes.
easeIn Elements leave — they accelerate as they exit, like they're being pulled away Dialogs closing, drawers sliding out, toasts dismissing. Exit animations should be faster than entrances so they don't delay the user's next action.
easeInOut Elements transform in place — symmetric acceleration and deceleration State changes that don't involve movement: hover color shifts, focus ring appearance, button press feedback, chip selection, progress bar fill. The element stays put but its appearance changes.

Enforcement

The motion spec has no teeth if developers can hardcode transition: 0.3s ease. Use the ESLint rule below to flag raw duration values in sx props and styled() calls.

DON'T — hardcoded values
// ✗ Raw duration — bypasses the spec sx={{ transition: 'all 0.3s ease' }} // ✗ Raw easing — not from the theme sx={{ transition: 'opacity 300ms cubic-bezier(0.4, 0, 0.2, 1)' }}
DO — theme values
// ✓ Theme duration token sx={{ transition: theme.transitions.create( ['opacity'], { duration: theme.transitions.duration.standard } )}} // ✓ Full shorthand with easing sx={{ transition: theme.transitions.create( ['background-color'], { duration: theme.transitions.duration.shorter, easing: theme.transitions.easing.easeInOut } )}}
// .eslintrc.js — flag raw transition durations in sx props rules: { 'no-restricted-syntax': ['error', { selector: 'Property[key.name="transition"] > Literal[value=/\d+ms|\d+s/]', message: 'Use theme.transitions.create() — no hardcoded durations.' }] } // PR checklist additions: // - [ ] No hardcoded transition durations (use theme.transitions) // - [ ] No hardcoded cubic-bezier values (use theme.transitions.easing)

Form Validation

● Stable

Validation is the highest-stakes interaction in Neighborly — participants are often anxious, low-literacy, or on mobile. Every validation decision has a direct impact on application completion rates. These patterns are mandatory, not optional.

Validation Timing

TriggerWhen to useNeighborly rule
On blur User leaves a field Default for all fields. Validate immediately on blur — don't wait for submit. Gives users early feedback without interrupting typing.
On change (debounced) Real-time feedback needed Use only for async checks (SSN uniqueness, email format) with a 400ms debounce. Never show errors while the user is still typing.
On submit Final gate before submission Always re-validate all fields on submit, even if blur validation already ran. Show an error summary at the top of the form.

Why a horizontal Stepper for application progress instead of a status chip?
A Stepper communicates where the participant is in the process and what comes next. A status chip (e.g., "Under Review") only communicates current state. The Stepper reduces anxiety by making the process transparent and predictable — critical for participants who may be in a stressful housing situation.
Why does the portal use warmer, lower-density design than the admin views?
Participants are not power users. Higher line height, larger font sizes, and more whitespace reduce cognitive load for users who may have lower digital literacy or be accessing the portal under stress. The warm color palette (#F5F7FA background, #3aab5c accents) is intentionally less clinical than the admin interface.
Why is the primary CTA 'Continue Application' rather than 'Edit'?
Action-oriented language reduces hesitation. 'Continue' implies progress and momentum; 'Edit' implies correction and error. For participants who may feel intimidated by the process, forward-looking language is more encouraging. This follows plain-language principles required by HUD for federally funded programs.
Alternative path: If a participant has completed all applications and is only waiting for decisions, replace the Stepper with a status timeline showing submitted → under review → decision dates.

Dynamic Announcements & Live Regions

Static WCAG compliance is necessary but not sufficient. The most common accessibility failures in dynamic web apps occur in live regions — content that changes after the page loads. Every dynamic state change that matters to a user must be announced to screen readers. Use the patterns below for all dynamic content in Neighborly.

VPAT reviewer note: WCAG 4.1.3 (Status Messages, AA) requires that status messages be programmatically determinable without receiving focus. All patterns below satisfy this criterion. This is the most commonly failed criterion in government procurement audits.
Pattern 1 — Toast / Snackbar Announcement

Toasts are transient. Use role="status" for non-critical messages (saved, submitted) and role="alert" for errors. Never use role="alert" for success messages — it interrupts screen reader flow unnecessarily.

// ✓ Success / info toast — role="status" (polite) <Snackbar open={open} autoHideDuration={5000}> <Alert severity="success" role="status" aria-live="polite"> Application #2024-0041 submitted successfully. </Alert> </Snackbar> // ✓ Error toast — role="alert" (assertive) <Snackbar open={open}> <Alert severity="error" role="alert" aria-live="assertive"> Submission failed. Please try again. </Alert> </Snackbar>
Pattern 2 — Form Validation Announcement

Inline field errors are announced when the field loses focus. For multi-field form submission errors, use an error summary Alert with role="alert" at the top of the form, and move focus to it programmatically.

// ✓ Inline field error — MUI helperText handles this automatically <TextField error={!!errors.ssn} helperText={errors.ssn?.message} // announced on blur inputProps={{ aria-describedby: "ssn-error" }} /> // ✓ Multi-field error summary — move focus to it on submit {hasErrors && ( <Alert severity="error" role="alert" tabIndex={-1} ref={errorSummaryRef} // call .focus() after submit > Please fix 3 errors before continuing. </Alert> )}
Pattern 3 — DataGrid Row Action Feedback

When a row action (Approve, Archive, Request Docs) completes, the DataGrid row updates silently. Add a visually hidden live region outside the grid to announce the result to screen readers.

// ✓ Hidden live region — place once in the page layout <Box aria-live="polite" aria-atomic="true" sx={{ position: 'absolute', left: '-9999px', width: '1px', height: '1px', overflow: 'hidden' }} ref={announcerRef} /> // ✓ On action complete, set the announcer text announcerRef.current.textContent = `Case CS-2024-1847 approved successfully.`;
Pattern 4 — Drawer Open/Close Focus Management

When a Drawer opens, focus must move into it. When it closes, focus must return to the trigger element. MUI Drawer handles this automatically when disableAutoFocus is not set. Always provide a visible close button as the first focusable element.

// ✓ MUI Drawer — focus management is automatic <Drawer anchor="right" open={open} onClose={handleClose} // DO NOT set disableAutoFocus or disableEnforceFocus > <IconButton onClick={handleClose} aria-label="Close case notes" autoFocus // first focusable element in drawer > <CloseIcon /> </IconButton> ... </Drawer>
Pattern 5 — Loading State Announcement

Skeleton screens and spinners are invisible to screen readers. Use aria-busy on the loading container and a visually hidden status message to communicate loading state.

// ✓ Loading container with aria-busy <Box aria-busy={isLoading} aria-label="Cases list" > {isLoading ? ( <> <Box sx={{ position: 'absolute', left: '-9999px' }}> Loading cases, please wait. </Box> <Skeleton variant="rectangular" height={400} /> </> ) : ( <DataGrid ... /> )} </Box>
Pattern 6 — Autocomplete Keyboard & Focus

MUI Autocomplete is built on ARIA 1.1 combobox pattern. It announces option count, selected value, and navigation automatically. The only override needed is a meaningful aria-label when there is no visible label.

// ✓ Autocomplete with full ARIA support <Autocomplete options={programs} getOptionLabel={(o) => o.name} renderInput={(params) => <TextField {...params} label="Grant Program" // visible label — no aria-label needed inputProps={{ ...params.inputProps, aria-autocomplete: "list", aria-haspopup: "listbox", }} /> } // Screen reader announces: "Grant Program, edit text, 12 options available" />

Brand Voice — Full Guide

Neighborly's voice is warm, direct, empowering, honest, and inclusive. Every string in the product — button labels, error messages, empty states, confirmation dialogs — is an opportunity to reinforce this voice. The Do/Don't examples below cover the four highest-stakes moments in the grant application experience.

Error Messages

✓ Do
"Please enter a valid ZIP code (e.g., 90210)."

Specific, tells the user exactly what to fix and gives an example.
✗ Don't
"Invalid input."

Vague and unhelpful. The user doesn't know what went wrong or how to fix it.

Empty States

✓ Do
"You haven't started any applications yet. Let's find programs you may qualify for."

Acknowledges the state, then gives a clear next step.
✗ Don't
"No applications found."

Reads like a database error. Leaves the user with no direction.

Success Confirmations

✓ Do
"Your application has been submitted. We'll review it and get back to you within 3–5 business days."

Confirms the action and sets a clear expectation.
✗ Don't
"Form submitted successfully."

Technically correct but cold. Doesn't tell the user what happens next.

Negative Eligibility Results

✓ Do
"Based on what you've shared, you may not qualify for these programs right now — but your situation can change. You can update your information at any time."

Softens the result, acknowledges impermanence, keeps the door open.
✗ Don't
"You are ineligible for all programs."

Blunt and final. Leaves the user feeling rejected with no hope or next step.

Denial is the highest-stakes moment in the participant journey. A person who has been waiting weeks — sometimes months — for a housing or utility grant decision deserves a different register than a generic error message. The copy patterns below apply to all denial and rejection states in the Participant Portal.

Legal note (HUD-funded programs): For programs funded under HUD guidelines, denial notices must include: (1) the specific reason(s) for denial, (2) the applicant's right to appeal, (3) the appeal deadline, and (4) contact information for the administering agency. These are not optional — they are regulatory requirements. Always include all four elements in any denial communication.

Tone Principles for Denial

Do

Don't

  • Use "unfortunately" — it reads as performative

AI UI Patterns New in v2.7

How to surface AI-generated content, suggestions, confidence scores, and AI-assisted actions in the Neighborly product. AI features must be transparent, correctable, and never presented as authoritative without human review.

Core Principles

Always label AI content
Any content generated or suggested by AI must be visually distinguished from human-authored content. Never present AI output as a system fact.
Always allow correction
Every AI suggestion must have an Accept, Edit, or Dismiss action. Caseworkers and admins must be able to override any AI output before it affects a case.
Show confidence honestly
Low-confidence outputs must be visually differentiated from high-confidence ones. Never hide uncertainty — especially for eligibility determinations.
Explain the basis
For consequential AI outputs (eligibility scores, fraud signals, risk flags), always provide a "Why?" disclosure — what data points drove the result.
Human in the loop
AI must not take irreversible actions autonomously. Approvals, denials, and disbursements always require a human action — AI can recommend, not decide.
Accessible by default
AI badges, confidence indicators, and suggestion chips must have accessible labels. Screen readers must announce AI-generated content as such.
