Motion & Animation

All animations respect the prefers-reduced-motion media query. When reduced motion is preferred, transitions are instant (0ms) or replaced with opacity fades. No animation should be the sole means of conveying information (WCAG 2.3.3).

Alerts & Banners

Alerts use role="alert" for errors and critical messages (announced immediately by screen readers), and role="status" for informational updates. Never use color alone to convey alert type.

Windows High Contrast Mode

All interactive elements are tested in Windows High Contrast Mode (Forced Colors). Buttons receive a 2px solid border. Focus rings use the system Highlight color. Gradients fall back to solid ButtonFace backgrounds.

@media (forced-colors: active) { button, [role="button"] { border: 2px solid ButtonText !important; forced-color-adjust: none; } :focus-visible { outline: 3px solid Highlight !important; outline-offset: 2px !important; } }

Tooltips & Popovers

Tooltips appear on hover and focus, providing brief supplementary information. They must not contain interactive elements. Popovers are richer overlays that can contain links, buttons, and structured content — they use role="tooltip" or role="dialog" accordingly.

If your household income is at or below 80% of the median income for your area, you may qualify for HUD-funded programs. This threshold varies by county and household size.

Brand Voice Guide

Neighborly's voice is warm, direct, and empowering. Every word in the product should reduce anxiety, not add to it. The people using our software are often navigating difficult circumstances — our language should feel like a trusted neighbor, not a government form. The following Do/Don't examples cover the four highest-impact moments in the product.

Scenario 1 — Error Messages
✓ Do
"We couldn't save your progress. Please check your connection and try again."

Explains what happened, who's responsible (not the user), and what to do next.

✗ Don't
"Error 500: Internal server error. Request failed."

Technical jargon creates anxiety and gives the user no path forward.

Scenario 2 — Empty States
✓ Do
"No applications yet. When you start one, it'll show up here so you can pick up right where you left off."

Reassures the user, explains what will happen, and reduces the feeling of a dead end.

✗ Don't
"No records found."

Cold and clinical. Gives no context and no next step.

Scenario 3 — Success Confirmations
✓ Do
"You're all set! Your application has been submitted. A caseworker will be in touch within 5–7 business days."

Celebrates the moment, confirms what happened, and sets clear expectations.

✗ Don't
"Submission successful. Application ID: #2024-0041."

Transactional and cold. Misses the opportunity to build trust and reduce post-submission anxiety.

Scenario 4 — Eligibility Results (Negative)
✓ Do
"Based on what you shared, you may not qualify for these programs right now — but circumstances change. We'll keep an eye out and let you know if new options open up."

Softens the blow, acknowledges impermanence, and keeps the door open.

✗ Don't
"You are ineligible for all programs."

Blunt and final. Leaves the user feeling rejected with no hope or next step.

Voice Principles at a Glance

PrincipleWhat it meansTest question
WarmSpeak like a trusted neighbor, not a bureaucratWould a kind neighbor say this?
DirectSay what you mean in as few words as possibleCan any word be removed without losing meaning?
EmpoweringAlways give the user a next step or a path forwardDoes this copy tell the user what to do next?
HonestNever overpromise; set accurate expectationsIs this statement 100% true?
InclusiveAvoid jargon, acronyms, and legal languageWould a 6th-grader understand this?

Readability & Measure

Measure is the typographic term for line length. Research consistently shows that lines between 45 and 75 characters produce the best reading speed and comprehension. Neighborly targets 60–75 characters for body text. This is enforced via max-width: 65ch on all long-form text containers.

✓ Optimal — 65ch max-width

Answer a few questions to find out which programs your household may qualify for. Your information is kept private and secure. No account is required to get started.

65 characters
✗ Too wide — no max-width constraint

Answer a few questions to find out which programs your household may qualify for. Your information is kept private and secure. No account is required to get started. The eye has to travel too far to return to the next line, causing fatigue and reducing comprehension for users with dyslexia or low literacy.

Readability Rules

RuleValueCSSWhy
Body text measure60–75 charactersmax-width: 65chOptimal reading speed and comprehension
Heading measure30–45 charactersmax-width: 40chHeadings scan better when short
Line height (body)1.6–1.65×line-height: 1.65Prevents lines from feeling cramped
Paragraph spacing≥2× font sizemargin-bottom: 1.5emWCAG 1.4.12 Text Spacing
Justified textNevertext-align: leftUneven word spacing harms readability
All-caps bodyNever—Reduces reading speed by ~13%

Loading States & Skeleton Screens

Loading states prevent double-submissions and communicate system activity. Skeleton screens replace empty containers while data loads, reducing perceived wait time by up to 40% compared to spinners alone. Every async action in the product must use one of these patterns.

Button Loading States

States

Add .btn-loading + aria-busy="true" on click. Remove both when the async action resolves.

// Toggle loading state on form submit submitBtn.addEventListener('click', async () => { submitBtn.classList.add('btn-loading'); submitBtn.setAttribute('aria-busy', 'true'); await submitApplication(); submitBtn.classList.remove('btn-loading'); submitBtn.removeAttribute('aria-busy'); });

Skeleton Screens

/* Skeleton utility classes */ .skeleton { background: linear-gradient(90deg, #E4EAF2 25%, #F0F4F8 50%, #E4EAF2 75%); background-size: 200% 100%; animation: skeleton-shimmer 1.4s ease-in-out infinite; } .skeleton-text { height: 14px; border-radius: 4px; } .skeleton-avatar { border-radius: 50%; } /* Always respect reduced motion */ @media (prefers-reduced-motion: reduce) { .skeleton { animation: none; background: #E4EAF2; } }

Responsive Data Tables

Complex data tables collapse into a card-per-row layout on viewports narrower than 640px. Each cell is labelled via a data-label attribute so the column header context is preserved without the <thead>. Caseworkers frequently access the system on tablets — this pattern ensures no data is lost or truncated.

Implementation note: Every <td> in a responsive table must include a data-label="Column Name" attribute. The CSS ::before pseudo-element reads this to render the label on mobile.
