Reconciliation note: MUI's default Typography variant="button" spec is uppercase, but the Neighborly theme overrides MuiButton.styleOverrides.root.textTransform: 'none'. In practice, all <Button> components render in sentence case. Use sentence case in all button labels. The variant="button" Typography variant is rarely used directly — it exists for non-button elements that need button-style text.

Line Length

Body text should be constrained to a maximum of 65 characters per line (max-width: 65ch). This prevents eye fatigue on wide screens and keeps reading comfortable for caseworkers who spend long sessions in the application.

Elevation

MUI's elevation system uses 25 levels (0–24). Neighborly uses a constrained set of four levels to maintain visual hierarchy without creating noise.

elevation=8

MUI <Dialog> interrupts the user for critical decisions. Use sparingly — only for destructive actions, confirmations, or complex forms that cannot be done inline. Always provide a clear dismiss path (Cancel button or close icon).

User Types

Neighborly has three distinct user types. All three share the same visual design system — the same components, colors, and patterns. What differs is the experience layer: what they see, what actions they can take, and how information is presented to them.

Participant
Participant
Community members applying for grant programs. May have limited digital literacy. Experience is optimized for clarity, guidance, and reducing anxiety.
  • Apply for grant programs
  • Upload required documents
  • Track application status
  • Receive notifications and decisions
  • View disbursement history
Contractor
Contractor
Service providers and vendors who deliver program services. Experience is task-focused — completing assigned work items and submitting deliverables.
  • View assigned service requests
  • Submit work completion reports
  • Upload invoices and receipts
  • Track payment status
  • Communicate with program staff
Admin
Admin / Caseworker
Program staff who manage the full grant lifecycle. Experience is data-dense and action-oriented — reviewing, approving, assigning, and monitoring.
  • Review and process applications
  • Assign applications to caseworkers
  • Approve or deny with documented reasoning
  • Manage program configuration
  • Generate reports and exports

Experience Differentiation

Application list My Applications — simple card list My Work Orders — task-focused All Applications — full DataGrid with filters
Status language "We're reviewing your application" "Work order assigned" Chip: Under Review, Pending Docs, etc.

Empty states are moments of opportunity — not failures. Always explain why the space is empty and provide a clear next action. Tailor the message to the user type. Brand Voice

Our voice is warm, clear, and human. We speak to participants as neighbors — not applicants, claimants, or case numbers. Every piece of copy should reduce anxiety, not add to it. When in doubt, ask: "Would a helpful neighbor say it this way?"

Voice Principles

PrincipleWhat it meansTest question
WarmAcknowledge the person, not just the transactionDoes this feel like it was written by a person who cares?
ClearPlain language, no jargon, 8th-grade reading levelWould someone unfamiliar with government programs understand this?
ActionableAlways tell the user what to do nextDoes the user know exactly what to do after reading this?
HonestDon't overpromise. Explain denials clearly and respectfullyIs this accurate, even if the news is hard?
InclusiveAvoid assumptions about income, housing, or family structureCould this alienate someone in a non-traditional situation?

MUI Theme Config

The Neighborly theme is created with createTheme() and passed to <ThemeProvider> at the app root. All palette values, typography, shape, and component overrides are defined here — never hardcoded in components.

Readability & Measure

Measure is the typographic term for line length. Research consistently shows that lines between 45 and 75 characters produce the best reading speed and comprehension. Neighborly targets 60–75 characters for body text, enforced via max-width: 65ch on all long-form text containers. This is especially important for participants who may have lower literacy levels.

Line Length Demo

✓ Optimal — 65ch max-width

Answer a few questions to find out which programs your household may qualify for. Your information is kept private and secure. No account is required to get started.

✗ Too wide — no max-width constraint

Answer a few questions to find out which programs your household may qualify for. Your information is kept private and secure. No account is required to get started. The eye has to travel too far to return to the next line, causing fatigue and reducing comprehension for users with dyslexia or low literacy.

Readability Rules

RuleValueCSS / MUI propWhy
Body text measure60–75 charactersmaxWidth: '65ch'Optimal reading speed and comprehension
Heading measure30–45 charactersmaxWidth: '40ch'Headings scan better when short
Line height (body)1.6–1.65×lineHeight: 1.65Prevents lines from feeling cramped
Paragraph spacing≥2× font sizemb: 2 (MUI spacing)WCAG 1.4.12 Text Spacing
Justified textNevertextAlign: 'left'Uneven word spacing harms readability
All-caps body textNever—Reduces reading speed by ~13%

Brand Voice — Full Guide

Neighborly's voice is warm, direct, empowering, honest, and inclusive. Every string in the product — button labels, error messages, empty states, confirmation dialogs — is an opportunity to reinforce this voice. The Do/Don't examples below cover the four highest-stakes moments in the grant application experience.

Error Messages

✓ Do
"Please enter a valid ZIP code (e.g., 90210)."

Specific, tells the user exactly what to fix and gives an example.
✗ Don't
"Invalid input."

Vague and unhelpful. The user doesn't know what went wrong or how to fix it.

Empty States

✓ Do
"You haven't started any applications yet. Let's find programs you may qualify for."

Acknowledges the state, then gives a clear next step.
✗ Don't
"No applications found."

Reads like a database error. Leaves the user with no direction.

Success Confirmations

✓ Do
"Your application has been submitted. We'll review it and get back to you within 3–5 business days."

Confirms the action and sets a clear expectation.
✗ Don't
"Form submitted successfully."

Technically correct but cold. Doesn't tell the user what happens next.

Negative Eligibility Results

✓ Do
"Based on what you've shared, you may not qualify for these programs right now — but your situation can change. You can update your information at any time."

Softens the result, acknowledges impermanence, keeps the door open.
✗ Don't
"You are ineligible for all programs."

Blunt and final. Leaves the user feeling rejected with no hope or next step.

Voice Principles at a Glance

PrincipleWhat it meansTest question
WarmSpeak like a trusted neighbor, not a bureaucratWould a kind neighbor say this?
DirectSay what you mean in as few words as possibleCan any word be removed without losing meaning?
EmpoweringAlways give the user a next step or a path forwardDoes this copy tell the user what to do next?
HonestNever overpromise; set accurate expectationsIs this statement 100% true?
InclusiveAvoid jargon, acronyms, and legal languageWould a 6th-grader understand this?

Styling: sx vs styled() vs Theme

MUI provides three styling approaches. Choosing the right one keeps the codebase consistent and avoids specificity conflicts. Use this decision guide for every new component.

Approach When to use When NOT to use Example
sx prop One-off layout tweaks, spacing overrides, responsive breakpoints on a single instance Styles that repeat across multiple components; complex pseudo-selectors <Box sx={{ mt: 2, display: 'flex' }}>
styled() Reusable styled variants of MUI components; complex selectors; animations Simple one-off spacing; anything that belongs in the theme const PipelineChip = styled(Chip)(({ theme }) => ({...}))
theme.components Global default overrides for a component (e.g. all Buttons pill-shaped); shared across the entire app Instance-specific overrides; anything that varies by context MuiButton: { styleOverrides: { root: { borderRadius: 20 } } }
DO — Use sx for layout, spacing, and responsive
// One-off layout on a single instance <Card sx={{ mt: 3, mb: 1, p: 2 }}> <CardContent sx={{ display: 'flex', gap: 2 }}> ... </CardContent> </Card>
DON'T — Repeat sx for shared styles
// Don't repeat this on every pipeline chip <Chip sx={{ borderRadius: '4px', fontWeight: 700, fontSize: '11px' }} /> <Chip sx={{ borderRadius: '4px', fontWeight: 700, fontSize: '11px' }} /> // Use styled() instead ↑
// CORRECT: Reusable pipeline status chip via styled() import { styled } from '@mui/material/styles'; import { Chip } from '@mui/material'; const StatusChip = styled(Chip)(({ theme, status }) => ({ borderRadius: '4px', fontWeight: 700, fontSize: '11px', ...(status === 'approved' && { backgroundColor: theme.palette.success.light, color: theme.palette.success.dark, }), ...(status === 'pending' && { backgroundColor: theme.palette.warning.light, color: theme.palette.warning.dark, }), })); // Usage: <StatusChip label="Approved" status="approved" size="small" />
